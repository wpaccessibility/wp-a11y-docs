Choosing an accessibility-ready theme is the cheapest decision you will ever make

When people set up a WordPress site, the theme is usually the first real decision they make. It sets the look, the layout, the feel. Most of the conversation around that choice is about design: does it match the brand, does it have the right demo content, does it support the block editor the way the team wants.

That is a reasonable place to start. But the theme decides something far more important than how the site looks. It decides the quality of the code underneath every page you will ever publish. And that code is now being read by a much larger audience than anyone planned for.

Here is the good news, and it is genuinely good news: the same choice that makes your site work for people using screen readers also makes it work for the AI agents and search engines that are becoming a serious part of your traffic. You do not have to pick between accessibility and discoverability. They run on the same foundation. Get the theme layer right, and you are set up for both on every page you publish afterward.

If you are not technical and that last paragraph already sounds like someone else’s problem, it is not. There is a separate, plainer article for you that skips the code and goes straight to what protects your traffic. This one goes under the hood, for the people who choose or build at the code layer. What the theme actually controls

With a theme, you are not writing the HTML yourself. The theme writes it for you. Every heading, every button, every navigation menu, every landmark on the page is generated by the theme’s templates and the blocks it ships with.

That is the part of your site you cannot reach from the editor. You can write perfect content, choose great color contrast, and order your headings correctly, and none of it will repair a theme that outputs a tangle of generic div elements with no semantic meaning. The structural layer belongs to the theme. You inherit whatever it gives you.

This is why the theme is a foundation decision, not a styling decision. A house with a solid foundation can be redecorated endlessly. A house built on a cracked one will keep showing problems no matter how nice the paint is.

Who actually reads your code now

For about twenty years, the audience for well-structured HTML was assistive technology. Screen readers, mainly. The argument for building accessibly was that people using those tools deserve to use your site, which has always been true and always been enough of a reason on its own.

What changed is that the audience grew. AI agents now read websites too. Search engines parse them more deeply than they used to. And here is the part that makes this easy to reason about: they all read your site through the same channel that a screen reader uses.

That channel is the accessibility tree. The browser builds it automatically from your HTML. It strips the page down to what matters for interaction, the headings, the buttons, the links, the form fields, the landmarks, and it records their labels, roles, and relationships. A screen reader reads that tree to describe your page to someone who cannot see it. An AI agent reads the same tree to understand what your page contains and what it can do there.

According to Nielsen Norman Group’s research on AI agents as users, reading the accessibility tree “costs a few thousand tokens to process (a fraction of what a screenshot requires)” and gives the agent more reliable information than trying to interpret a picture of the page. Roughly four thousand tokens for the tree against fifty thousand for a screenshot of the same page. It is cheaper and faster, and it gives a more accurate result.

But the tree is only as good as the HTML it is built from. As the No Hacks guide on how AI agents see websites puts it, “use semantic elements, and the browser generates a useful accessibility tree automatically.” Skip them, and the tree is sparse and misleading. The browser can only describe what you actually marked up.

So the thing that makes your site usable for a person with a screen reader is the exact same thing that makes it readable, and cheap, for an agent to use. It is one decision that serves both audiences, with nothing traded away to get there.

Why accessible structure helps your ranking

The SEO connection follows from the same mechanism, and it is worth understanding rather than taking on faith.

Agents and search engines increasingly favor content they can parse reliably and cheaply. As Matthew Stephens wrote in his March 2026 analysis of AI agents and accessibility, “agents optimized for token efficiency will perform better on well-structured interfaces.” A site with clean semantic structure is one an agent can read confidently and summarize correctly. A site that forces an agent to guess produces weak summaries, and weak summaries send people who click through straight back to the search results, which is exactly the kind of signal that hurts you.

Accessible structure also happens to be everything good SEO has always asked for. Proper heading hierarchy, descriptive link text, meaningful landmarks, content that lives in the initial HTML rather than appearing only after JavaScript runs. Search engines have wanted this for years. Accessibility and SEO were never separate disciplines pulling in different directions. They ask for the same structure for the same underlying reason: the content needs to be understandable to something that is not looking at the screen.

Jono Alderson put the shift plainly: your site “isn’t just a visual experience anymore. It’s an interface. An API. A dataset.” The sites that present that interface clearly are the ones a growing number of automated readers will be able to use, and prefer.

What this means if you choose themes

If you are the person selecting a theme, you have a shortcut, and it just got more reliable.

The WordPress.org theme directory has an accessibility-ready tag. To carry it, a theme has to pass a review against the Accessibility Team’s requirements, things like keyboard navigation, proper heading structure, labeled controls, and meaningful landmarks. The structural fundamentals that produce a good accessibility tree are exactly what that review checks for.

That tag just became a stronger signal than it has been in years. In May 2026 the Accessibility Team published the first full rewrite of the accessibility-ready requirements since 2012, the first update in fourteen years. Themes carrying the tag are being retested against the new requirements, and authors have until the end of June 2026 to comply or lose the tag. So a theme that holds the accessibility-ready tag after that retest has been checked against current standards, not standards from over a decade ago.

Starting from an accessibility-ready theme does not mean your work is done. You still have to test, because content choices and plugins can introduce problems the theme never had. But it means your foundation is sound, and you are not trying to retrofit semantic structure onto a theme that was never built to provide it. That retrofit is the expensive path. Starting from a good foundation is the cheap one.

What this means if you build themes

If you build themes, this is an opportunity worth taking seriously, because the value of what you produce just went up without the requirements changing.

The semantic HTML that produces a good accessibility tree, the proper heading structure, the real button elements, the labeled form fields, the landmark regions, now serves more audiences than it used to. It serves the people using assistive technology, which was always the point. It serves the search engines indexing your users’ sites. And it serves the AI agents that increasingly decide which sites are cheap enough to bother with. If you already build this way, your existing skill reaches those new audiences without changing anything about how you work. If you do not yet, this is the moment it becomes worth learning, because the same effort now pays off in more places than it did even a year ago.

Pursuing the accessibility-ready tag, and keeping it through the new retest, is a way to make that quality visible and findable. With the rewritten requirements, the tag now reflects a current, meaningful standard. A theme that carries it tells everyone choosing a theme that the foundation is solid, and increasingly that is a competitive advantage rather than a nice extra.

The structure that earns the tag is the same structure that makes every site built on your theme easier for agents and search engines to read. You are building the foundation thousands of sites will stand on. Building it well has never paid off in more directions at once.

A word for plugins, in both directions

Themes get most of the attention here, but plugins output code too. A plugin that injects content, a form, a slider, a checkout step, is adding HTML to pages it does not own. If that HTML is inaccessible, it can undermine an otherwise solid theme.

It is genuinely disappointing when a carefully chosen accessibility-ready theme runs into trouble because a plugin drops unlabeled form fields or clickable divs into the page. The site owner did the right thing at the theme layer and still inherited a problem they cannot see and cannot fix from the editor.

So the same reasoning applies a level down. If you choose plugins, it is fair to ask the same question you ask of a theme: does this output accessible, semantic markup? And if you build plugins, the structure that makes your output accessible is the structure that makes it readable to every agent and search engine that meets it. A plugin that respects the accessibility tree is a plugin that does not quietly damage the discoverability of every site it is installed on. That is a strong thing to be able to say about your work.

The foundation pays off in every direction

For most of its history, the case for building accessible WordPress sites rested on one audience: people using assistive technology. That was always reason enough.

What is new is that the same structural decisions now pay off in directions nobody was counting on. Accessible themes and plugins produce the clean, semantic code that search engines reward and that AI agents can read cheaply and reliably. The accessibility-ready tag, freshly modernized for the first time in fourteen years, is a practical shortcut to that foundation for the people choosing, and a meaningful mark of quality for the people building.

Choose an accessibility-ready theme and you are building on solid ground from the first page. Build accessibility-ready themes and plugins and you are handing thousands of site owners that solid ground. The structure that includes every human being also turns out to be the structure that keeps a site visible as the web fills up with readers that were never looking at the screen in the first place.

That is a rare kind of decision. It costs you nothing extra and it pays off for everyone who touches the site after you.
